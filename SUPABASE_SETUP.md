# Supabase Integration Setup Guide

This guide explains how to connect and run the **Pace** synchronization layer with your Supabase database.

---

## 1. Required Environment Variables

Create a `.env` file at the root of your project (next to `package.json`) and configure your Supabase project endpoints:

```env
VITE_SUPABASE_URL=https://your-project-id.supabase.co
VITE_SUPABASE_PUBLISHABLE_KEY=your-anon-key
```

*Note: Restart Vite after creating or editing the `.env` file.*

---

## 2. Applying the Schema

To initialize the database tables, RLS rules, and access policies in your Supabase dashboard:

1. Open your project on the [Supabase Dashboard](https://supabase.com).
2. Go to the **SQL Editor** view in the sidebar.
3. Open `supabase/schema.sql` from your local Pace project workspace.
4. Copy the entire contents of the file.
5. Paste it into the Supabase SQL editor and click **Run**.

This script will:
* Create all 8 mirrored tables with `deleted_at` soft delete fields.
* Add indexes and reference keys.
* Enable Row Level Security (RLS) on each table.
* Establish secure SELECT, INSERT, UPDATE, and DELETE policies allowing only authenticated users to access their own data.

---

## 3. How Synchronization Works

The sync engine uses an **offline-first** design. The application reads and writes exclusively to the local IndexedDB database using **Dexie.js**. This ensures instant response times and offline functionality.

### Sync Cycle Overview
When `syncAll()` is executed:
1. **Push Phase**: The manager reads all local entries from Dexie for all 8 tables and upserts them to Supabase, matching the active user's `user_id`.
2. **Pull Phase**: The manager queries Supabase for the active user's remote records. It upserts new and modified items into Dexie and deletes local items that are no longer present in Supabase (supporting remote deletion propagation).

### Composite Key Design
To allow multiple users to sync their databases without conflicts:
* Every table has a composite primary key: `(user_id, id)`.
* `id` corresponds directly to the auto-incrementing local ID generated by Dexie.
* This allows each user to have their own record `1`, `2`, `3` independently.
* Relational joins (foreign keys) are checked against the compound pair `(user_id, project_id)` or `(user_id, habit_id)` to preserve referential integrity.

## 4. How to Use the Sync Manager

To perform a sync, import `syncAll` from `src/db/syncManager`:

```typescript
import { syncAll } from './db/syncManager';

const stats = await syncAll();
if (stats.success) {
  console.log('Synchronized successfully at', stats.timestamp);
} else {
  console.error('Synchronization failed:', stats.error);
}
```

## 5. Reactive Sync Status Hook

Monitor the real-time state of the sync engine inside React components using `useSyncStatus()`:

```typescript
import { useSyncStatus } from '../hooks/useSyncStatus';

function SyncIndicator() {
  const status = useSyncStatus();

  return (
    <div className={`status-badge ${status.toLowerCase()}`}>
      Sync Status: {status}
    </div>
  );
}
```

### Possible Sync States:
* `Idle`: All local mutations have synced; engine is waiting.
* `Syncing`: Pushing local mutations or pulling remote updates.
* `Offline`: Navigator reports no internet connection; sync processes are paused.
* `Waiting for Retry`: A sync operation failed; waiting for exponential backoff before next attempt.
* `Conflict Resolved`: A newer remote timestamp replaced an older local record.
* `Error`: Sync cycle encountered an unexpected error.

## 6. How to Authenticate Users

Use the utility functions inside `src/lib/auth.ts`:

```typescript
import { signUp, signIn, signOut, getCurrentUser } from './lib/auth';

// Register User
const user = await signUp('user@example.com', 'password123');

// Login User
const user = await signIn('user@example.com', 'password123');

// Logout User
await signOut();

// Get active session user info
const user = await getCurrentUser();
```

## Health Mode tables

Health mode (exercise, water, sleep, diet, weight, steps, goals) syncs through
seven additional tables. To enable it, run `supabase/migration_health.sql` in
the Supabase SQL editor. Until the migration is applied, health data still
works fully offline on each device — it just won't sync across devices, and
the console will log a warning per health table during sync.
